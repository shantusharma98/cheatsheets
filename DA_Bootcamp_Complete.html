E1 You have 50,000 rows of data. How do you quickly find and select all blank cells in column B to fill them with "Unknown"? Medium▼
Answer
Select column B → Ctrl+G (Go To) → Special → Blanks → OK. All blank cells selected. Type "Unknown" → Ctrl+Enter (fills ALL selected cells simultaneously). This technique works on any column and is much faster than filtering.
💡 Ctrl+Enter vs Enter: Regular Enter moves to next cell. Ctrl+Enter fills all selected cells with the same value.
E2 What is the difference between "Delete" key and right-click → Delete on a row? Trick question commonly asked! TRICK▼
Answer
Common confusion: Delete key and Delete row are completely different operations
Delete key only clears the CONTENTS of a cell (the cell still exists, row count unchanged).

Right-click → Delete → Entire Row removes the row entirely and shifts rows up — row count decreases by 1.

This matters when formulas reference specific row numbers, or when you have data below that needs to shift up.

E3 Column A has full names like "Smith, John" (Last, First format). Write formulas to extract First Name and Last Name into separate columns. Medium▼
Answer
-- Last Name (everything before the comma)
=LEFT(A2, FIND(",", A2) - 1)

-- First Name (everything after ", ")
=MID(A2, FIND(",", A2) + 2, LEN(A2))

-- Or combine back: "John Smith"
=TRIM(MID(A2,FIND(",",A2)+1,99)) & " " & LEFT(A2,FIND(",",A2)-1)
E4 Calculate the number of working days between two dates excluding weekends AND a list of holidays in E2:E15. Hard▼
Answer
=NETWORKDAYS(start_date, end_date, E2:E15)
-- NETWORKDAYS automatically excludes Saturdays & Sundays
-- Third argument = holiday range to also exclude
-- Returns count inclusive of both start and end dates

-- For custom weekend (e.g., Fri+Sat instead of Sat+Sun):
=NETWORKDAYS.INTL(start_date, end_date, 7, holidays)
-- weekend codes: 1=Sat+Sun, 7=Fri+Sat, 11=Sun only, etc.
E5 You have a column of numbers. Write a single formula (no helper columns) to count how many values are above average. Medium▼
Answer
=COUNTIF(A2:A100, ">"&AVERAGE(A2:A100))
-- The & joins ">" with the calculated average value
-- Note: criteria must be text + number concatenated, not just >AVERAGE()

-- Alternative using SUMPRODUCT (works for complex conditions):
=SUMPRODUCT((A2:A100 > AVERAGE(A2:A100)) * 1)
⚠️ Common mistake: =COUNTIF(A2:A100, >AVERAGE(A2:A100)) will ERROR. The criterion must be a string: ">"&AVERAGE()
E6 TRICK: What does =A1="" return if A1 contains a space " "? What about if A1 is truly empty? TRICK▼
Answer
Many people assume =A1="" catches all "empty" cells — it doesn't!
If A1 = " " (space): =A1="" returns FALSE — a space is a character, not empty.
If A1 is truly empty: =A1="" returns TRUE

Better approach: =TRIM(A1)="" catches spaces too.
Even better: =ISBLANK(A1) only returns TRUE for genuinely empty cells (not formula cells that return "").
E7 Use SUMPRODUCT to count unique values in range A2:A100 (without removing duplicates). Hard▼
Answer
=SUMPRODUCT(1/COUNTIF(A2:A100, A2:A100))
-- How it works: COUNTIF returns how many times each value appears
-- 1/count gives: if value appears 3 times → 1/3 + 1/3 + 1/3 = 1
-- So each unique value contributes exactly 1 to the total

-- If range has blanks, protect against divide-by-zero:
=SUMPRODUCT((A2:A100<>"")/COUNTIF(A2:A100, A2:A100&""))

E8 VLOOKUP returns the WRONG value even though the data exists. Name 5 common reasons this happens. TRICK▼
Answer — 5 Silent VLOOKUP Killers
This is a classic interview question. Most people only know 1-2 reasons.
1. Last argument missing / TRUE — Default is approximate match (sorted data assumed). Always use FALSE/0 for exact match.
2. Numbers stored as text — "123" (text) ≠ 123 (number). VLOOKUP won't find it. Fix: multiply by 1 or use VALUE().
3. Trailing/leading spaces — "Smith " ≠ "Smith". Fix: wrap lookup value in TRIM().
4. Lookup column not leftmost — VLOOKUP can only look RIGHT from column 1. Use INDEX+MATCH instead.
5. Range not locked ($) — When copying formula down, the table_array shifts. Always use $A$2:$D$100.
E9 Two-way lookup: find the value at the intersection of a row label (in A column) and a column header (in row 1). Use INDEX+MATCH+MATCH. Hard▼
Answer
-- Lookup value: row label in E2, column label in F2
-- Data table: B2:D10, row headers in A2:A10, col headers in B1:D1
=INDEX(B2:D10,
    MATCH(E2, A2:A10, 0),   -- finds the row
    MATCH(F2, B1:D1, 0)     -- finds the column
)

-- XLOOKUP version (cleaner):
=XLOOKUP(E2, A2:A10, XLOOKUP(F2, B1:D1, B2:D10))
💡 Two-way lookup with XLOOKUP nests one inside the other: inner XLOOKUP returns the matching column, outer XLOOKUP finds the row.
E10 TRICK: What's wrong with =VLOOKUP(A2,B:D,3,FALSE) — will it work? Is it good practice? TRICK▼
Answer
It WORKS but is bad practice — whole column references are inefficient
It works — using B:D as the table_array is valid syntax.

But it's bad practice because: It scans the ENTIRE column (1,048,576 rows) instead of just your data range. On large files this causes serious slowdowns. Excel also includes the header row in the lookup, which can cause false matches.

Better: =VLOOKUP(A2, $B$2:$D$1000, 3, FALSE) — bounded range, locked with $.

Interview Question
Why should you almost never use a 3D chart in a professional report?
3D charts distort perception — the depth adds visual complexity that makes it hard to accurately read values. The bars/slices at the back appear smaller than those at the front due to perspective, misleading the reader. They also print poorly and are inaccessible for people with depth perception issues. Always use 2D charts for accuracy and clarity in analytics.

E12 TRICK: Your pivot table shows incorrect totals — the Grand Total doesn't match the sum of visible items. Why? TRICK▼
Answer
Pivot Grand Totals can seem wrong — here's why they're actually correct
This happens when you use a Calculated Field that performs a non-additive operation (like averages or ratios). The Grand Total recalculates the formula on all underlying data — it does NOT sum the displayed row values.

Example: A calculated field = Revenue/Units. The Grand Total = Total Revenue / Total Units (correct), NOT the sum of each row's ratio (which would be wrong).

Fix: If you need a sum of ratios, use a helper column in your source data, not a calculated field.

Excel Interview & Trick Questions

TRICKINTERVIEW
Trick Question
What does VLOOKUP return if the match_type argument is TRUE (or omitted) and the data is NOT sorted?
Wrong answer: "It finds the exact match anyway." Reality: it returns WRONG data silently
It returns an incorrect value with no error. Approximate match (TRUE) assumes the first column is sorted ascending. On unsorted data it uses binary search and stops early, returning whatever it finds. The dangerous part: no error message — you get confidently wrong numbers. This is why =VLOOKUP(x, range, col, FALSE) should be your default.
Trick Question
=SUM(IF(A1:A10>5, A1:A10)) — will this work? What's the catch?
In older Excel this requires Ctrl+Shift+Enter. In 365 it just works.
In Excel 2019 and older, this is an array formula and MUST be entered with Ctrl+Shift+Enter (it shows {curly braces} around it). Pressing just Enter gives the wrong result (only evaluates first cell).

In Excel 365 / Excel 2021, dynamic arrays handle this automatically — just press Enter. However, the modern equivalent is simply =SUMIF(A1:A10,">5").
Interview Question
What is Power Query and how is it different from regular Excel formulas?
Power Query (Data → Get & Transform) is a data transformation engine built into Excel. Unlike formulas which transform one cell at a time, Power Query transforms entire datasets as a pipeline of steps. Key differences:
• Repeatable: Steps are recorded — refresh imports and re-applies all transformations
• Scale: Handles millions of rows (beyond Excel's 1M row limit when loading to data model)
• No formulas in cells: Transformations happen before data lands in the sheet
• M language: Each step generates M code which can be edited directly
Use it for: combining multiple files, unpivoting data, complex cleaning, merging tables from different sources.
Interview Question
Explain the difference between COUNTA, COUNT, and COUNTBLANK. When would each give different results?
COUNT — counts only numeric values (ignores text, blanks, errors)
COUNTA — counts all non-blank cells (numbers, text, errors — everything except truly empty cells)
COUNTBLANK — counts empty cells AND cells containing formula that returns ""

They differ when: column has mixed types (text + numbers), cells have formulas returning "", or cells look empty but contain spaces.

SQL Fundamentals — SELECT, WHERE, ORDER BY

CONCEPTMilestone 4

SQL Query Execution Order (Not Written Order)

-- WRITTEN order:  SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
-- EXECUTED order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

-- Why this matters: you CANNOT use SELECT aliases in WHERE or GROUP BY
-- This is WRONG:
SELECT Revenue * 1.1 AS NewRev
FROM Sales
WHERE NewRev > 1000;  -- ERROR: WHERE runs before SELECT, alias not yet defined

-- This is CORRECT:
SELECT Revenue * 1.1 AS NewRev
FROM Sales
WHERE Revenue * 1.1 > 1000;  -- repeat the expression in WHERE
Q1 TRICK: Can you use a column alias in a GROUP BY? What about in ORDER BY? Why the difference? TRICK▼
Answer
Most candidates say "no" to both — but ORDER BY is actually YES in most databases!
GROUP BY: Cannot use aliases in standard SQL (executes before SELECT). However, MySQL/MariaDB allow it as an extension.

ORDER BY: YES, you can use aliases! ORDER BY executes AFTER SELECT, so the alias is already defined. This works in all major databases (PostgreSQL, MySQL, SQL Server, SQLite).

SELECT revenue * 1.1 AS adj_rev FROM sales ORDER BY adj_rev DESC; ✅
Q2 Write a query to find all Northwind products where the product name contains "ch" (case-insensitive) and price is between $10-$50. Easy▼
Answer
SELECT ProductID, ProductName, UnitPrice
FROM   Products
WHERE  LOWER(ProductName) LIKE '%ch%'
  AND  UnitPrice BETWEEN 10 AND 50
ORDER BY UnitPrice;
💡 BETWEEN is inclusive on both ends: BETWEEN 10 AND 50 = UnitPrice >= 10 AND UnitPrice <= 50
Q3 TRICK: What is NULL? Is NULL = NULL true in SQL? How do you properly check for NULL? TRICK▼
Answer
Almost everyone says NULL = NULL is TRUE. It's FALSE — and this breaks many queries silently.
NULL represents the absence of a value — it's unknown, not zero or empty string.

NULL = NULL → returns NULL (not TRUE!) This is three-value logic: TRUE, FALSE, or NULL.

This means: WHERE col = NULL will NEVER return rows, even if col has NULLs.

Correct way: WHERE col IS NULL or WHERE col IS NOT NULL

Also: Any arithmetic with NULL returns NULL. 5 + NULL = NULL. Any comparison with NULL returns NULL (falsy). This is why COUNT(*) ≠ COUNT(col) when col has NULLs.
Q4 TRICK: What is the difference between COUNT(*), COUNT(col), and COUNT(DISTINCT col)? TRICK▼
Answer
COUNT(*): counts ALL rows including NULLs and duplicates
COUNT(col): counts rows where col is NOT NULL (ignores NULLs)
COUNT(DISTINCT col): counts unique non-null values in col

Example — if col has values [1, 1, 2, NULL, NULL]:
COUNT(*) = 5, COUNT(col) = 3, COUNT(DISTINCT col) = 2

Aggregation — GROUP BY, HAVING, CASE WHEN

PRACTICEMilestone 5
Q5 Find the top 3 categories by average product price, but only include categories with more than 5 products. Medium▼
Answer
SELECT
    c.CategoryName,
    COUNT(p.ProductID)          AS ProductCount,
    ROUND(AVG(p.UnitPrice), 2)  AS AvgPrice
FROM   Categories c
JOIN   Products p ON c.CategoryID = p.CategoryID
GROUP BY c.CategoryName
HAVING COUNT(p.ProductID) > 5
ORDER BY AvgPrice DESC
LIMIT  3;
Q6 Classify each order as 'Small' (<100), 'Medium' (100-500), 'Large' (>500) based on total order value. Count orders in each class. Medium▼
Answer
SELECT
    CASE
        WHEN OrderTotal <  100 THEN 'Small'
        WHEN OrderTotal <= 500 THEN 'Medium'
        ELSE                         'Large'
    END AS OrderCategory,
    COUNT(*) AS OrderCount
FROM (
    SELECT o.OrderID,
           SUM(od.Quantity * od.UnitPrice * (1-od.Discount)) AS OrderTotal
    FROM   Orders o
    JOIN   [Order Details] od ON o.OrderID = od.OrderID
    GROUP BY o.OrderID
) t
GROUP BY OrderCategory
ORDER BY OrderCount DESC;
Q7 TRICK: You have a table with salary data. Write a query to find the second highest salary WITHOUT using LIMIT/OFFSET or any window function. TRICK▼
Answer
-- Method 1: Subquery with MAX (classic)
SELECT MAX(Salary) AS SecondHighest
FROM   Employees
WHERE  Salary < (SELECT MAX(Salary) FROM Employees);

-- Method 2: Self-join (old school)
SELECT MAX(e1.Salary)
FROM   Employees e1
WHERE  e1.Salary < (SELECT MAX(e2.Salary) FROM Employees e2);

-- Method 3: DENSE_RANK (modern, handles ties properly)
SELECT Salary FROM (
    SELECT Salary, DENSE_RANK() OVER (ORDER BY Salary DESC) rk
    FROM Employees
) t WHERE rk = 2;
⚠️ Methods 1 & 2 fail on ties — if two people share the highest salary, they return the same person's salary. Method 3 (DENSE_RANK) is the safest.

Q8 TRICK: You write a LEFT JOIN but get the same results as an INNER JOIN. What likely went wrong? TRICK▼
Answer
This is a very common real-world bug that wastes hours of debugging time
You put a filter on the RIGHT table in the WHERE clause instead of the ON clause.

This converts a LEFT JOIN into an INNER JOIN silently:
-- WRONG: WHERE converts LEFT JOIN to INNER JOIN
SELECT * FROM Customers c
LEFT JOIN Orders o ON c.CustomerID = o.CustomerID
WHERE  o.ShipCountry = 'Germany';  -- NULL rows get filtered out!

-- CORRECT: Filter in ON clause preserves NULL rows
SELECT * FROM Customers c
LEFT JOIN Orders o
    ON  c.CustomerID = o.CustomerID
    AND o.ShipCountry = 'Germany';  -- NULL rows preserved
Q9 Find customers who ordered in 1996 but NOT in 1997 (i.e., they churned between years). Use two different approaches. Hard▼
Answer
-- Method 1: LEFT JOIN Anti-Join (most efficient)
SELECT DISTINCT o96.CustomerID
FROM   Orders o96
LEFT  JOIN Orders o97
    ON  o96.CustomerID = o97.CustomerID
    AND strftime('%Y', o97.OrderDate) = '1997'
WHERE  strftime('%Y', o96.OrderDate) = '1996'
  AND  o97.CustomerID IS NULL;

-- Method 2: NOT IN subquery
SELECT DISTINCT CustomerID FROM Orders
WHERE  strftime('%Y', OrderDate) = '1996'
  AND  CustomerID NOT IN (
      SELECT DISTINCT CustomerID FROM Orders
      WHERE  strftime('%Y', OrderDate) = '1997'
  );
⚠️ NOT IN is dangerous if the subquery can return NULLs — any NULL makes the entire NOT IN return FALSE. Prefer the LEFT JOIN method.
Q10 TRICK: What happens when you JOIN on a column that has duplicate values in BOTH tables? How many rows does the result have? TRICK▼
Answer — The Cartesian Trap
This is called a "fan-out" or "join explosion" — and it silently inflates your numbers
You get a Cartesian product for the duplicate values.

Example: Table A has 3 rows with CustomerID=5. Table B has 2 rows with CustomerID=5. The JOIN produces 3×2 = 6 rows for CustomerID=5.

This causes inflated SUM values — a classic analytics bug. You'll sum revenue 3x when you should sum it once.

Prevention: Before joining, check for duplicates in the join key: SELECT key, COUNT(*) FROM table GROUP BY key HAVING COUNT(*) > 1. Use DISTINCT or aggregate to deduplicate before joining.

Subqueries, CTEs & Derived Tables

PRACTICEMilestone 7
Q11 Using a CTE chain: find the top 5 customers by revenue, then show what % of total company revenue each represents. Hard▼
Answer
WITH CustomerRev AS (
    SELECT
        c.CompanyName,
        ROUND(SUM(od.Quantity*od.UnitPrice*(1-od.Discount)),2) AS Rev
    FROM Customers c
    JOIN Orders o ON c.CustomerID = o.CustomerID
    JOIN [Order Details] od ON o.OrderID = od.OrderID
    GROUP BY c.CompanyName
),
TotalRev AS (
    SELECT SUM(Rev) AS Total FROM CustomerRev
)
SELECT
    cr.CompanyName,
    cr.Rev,
    ROUND(cr.Rev * 100.0 / tr.Total, 2) AS PctOfTotal
FROM CustomerRev cr, TotalRev tr
ORDER BY cr.Rev DESC
LIMIT 5;
Q12 Recursive CTE: generate a number series from 1 to 12 (useful for generating date spines or calendar tables). Hard▼
Answer
WITH RECURSIVE nums(n) AS (
    SELECT 1                -- anchor: starting value
    UNION ALL
    SELECT n + 1           -- recursive: add 1 each time
    FROM   nums
    WHERE  n < 12          -- termination condition
)
SELECT n FROM nums;
-- Output: 1, 2, 3, ..., 12
-- Great use: generate all months, fill missing date gaps

Q13 TRICK: What's the difference between SUM() OVER() with no ORDER BY vs with ORDER BY? Many candidates get this wrong. TRICK▼
Answer
Adding ORDER BY to a window aggregate completely changes what it calculates
-- Without ORDER BY: same value for all rows = Grand Total
SUM(Revenue) OVER (PARTITION BY Region)
-- Returns: total revenue for that region in every row

-- With ORDER BY: RUNNING TOTAL (cumulative sum)
SUM(Revenue) OVER (PARTITION BY Region ORDER BY OrderDate)
-- Returns: cumulative revenue up to current row within region
⚠️ The default frame with ORDER BY is "ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW" — this gives running total, not the full window sum.
Q14 For each employee, show: their monthly sales, previous month's sales (LAG), and % change. Flag months where growth dropped below -10%. Hard▼
Answer
WITH MonthlySales AS (
    SELECT
        e.FirstName || ' ' || e.LastName     AS Employee,
        strftime('%Y-%m', o.OrderDate)      AS Month,
        ROUND(SUM(od.Quantity*od.UnitPrice),2) AS Sales
    FROM Employees e
    JOIN Orders o ON e.EmployeeID = o.EmployeeID
    JOIN [Order Details] od ON o.OrderID = od.OrderID
    GROUP BY Employee, Month
)
SELECT
    Employee, Month, Sales,
    LAG(Sales) OVER (PARTITION BY Employee ORDER BY Month) AS PrevMonth,
    ROUND((Sales - LAG(Sales) OVER (PARTITION BY Employee ORDER BY Month))
          * 100.0 / NULLIF(LAG(Sales) OVER (PARTITION BY Employee ORDER BY Month),0),1) AS GrowthPct,
    CASE WHEN (Sales - LAG(Sales) OVER(PARTITION BY Employee ORDER BY Month)) * 100.0
              / NULLIF(LAG(Sales) OVER(PARTITION BY Employee ORDER BY Month),0) < -10
         THEN '⚠ DROP' ELSE '' END AS Alert
FROM MonthlySales;
💡 NULLIF(x, 0) prevents division by zero — returns NULL when x=0, and NULL/anything = NULL (not an error)

Advanced SQL — Indexes, Optimization, EXPLAIN

ADVANCED

Query Optimization Checklist

  • Use indexes on JOIN columns and WHERE-filtered columns — dramatically speeds up queries
  • Avoid SELECT * — read only columns you need; reduces I/O
  • Filter early — put the most selective WHERE conditions first
  • Avoid functions on indexed columns in WHERE: WHERE YEAR(date) = 2024 can't use an index on date — instead use WHERE date BETWEEN '2024-01-01' AND '2024-12-31'
  • EXISTS vs IN: EXISTS stops at first match (faster for large subqueries); IN evaluates all values
  • EXPLAIN / EXPLAIN ANALYZE: shows query execution plan — look for "Full Table Scan" (bad) vs index usage (good)
Trick Question
Is it faster to filter with WHERE before or after a JOIN? Why?
Most people say "it doesn't matter — the optimizer handles it." Partially true, but not always.
Filtering BEFORE joining (in a subquery or CTE) reduces the rows in the join operation. While modern query optimizers often push filters down automatically ("predicate pushdown"), complex queries with multiple CTEs or views may not be optimized. Explicitly filtering early (or applying indexes to the filter columns) is always safer and more predictable. Rule of thumb: join on as small a dataset as possible.

SQL Interview & Trick Questions

TRICKINTERVIEW
Trick Question
DELETE, TRUNCATE, DROP — what are the differences? Which can be rolled back?
DELETE: Removes specific rows (with WHERE) or all rows. Logs each row deletion. Can be rolled back (in a transaction). Triggers fire.
TRUNCATE: Removes ALL rows. Much faster (minimal logging). Cannot be rolled back in most DBs. No triggers. Resets identity/auto-increment.
DROP: Removes the entire table (structure + data). Cannot be rolled back. Requires re-creating the table.

Memory trick: Delete = selective surgeon, Truncate = power wash the table, Drop = demolish the building.
Trick Question
What is the difference between UNION and UNION ALL? When would using UNION ALL be wrong?
People know UNION removes duplicates, but can't explain WHEN that's dangerous vs desirable
UNION: combines result sets and removes duplicate rows (slower — performs a sort/distinct).
UNION ALL: combines result sets keeping ALL rows including duplicates (faster).

UNION ALL is WRONG when: you're combining two queries that naturally produce overlapping data and you need unique results (e.g., combining two lists of customer IDs where some customers appear in both).

UNION is WRONG when: you're doing aggregations — using UNION removes legitimate duplicate values from your aggregation dataset, corrupting results. Always use UNION ALL for performance and correctness unless you specifically need deduplication.
Interview Question
Explain the difference between a view, a materialized view, and a CTE. When would you use each?
VIEW: A saved SQL query (virtual table). Executes the query every time you SELECT from it. No data stored. Always shows current data. Use for: simplifying complex joins, providing a stable interface, security (hide raw tables).

MATERIALIZED VIEW: A view that actually stores the result set on disk. Must be refreshed manually or on schedule. Much faster to query. Use for: expensive aggregations you query frequently, reporting dashboards.

CTE: Temporary, query-scoped only. Exists only for the duration of one query. Cannot be indexed. Use for: breaking up complex queries into readable steps, recursive queries.

D1 TRICK: What does CALCULATE(SUM(Sales[Revenue])) with NO filter arguments do? Is it the same as SUM(Sales[Revenue])? TRICK▼
Answer
Most people say "yes, they're the same." They're almost the same but have a critical difference with context transition.
In most cases they return the same value — the current filter context's sum.

The difference: CALCULATE always performs a context transition. If used inside a row context (like a calculated column or an iterator like SUMX), CALCULATE converts the row context into an equivalent filter context. This means it will apply filters for the current row's values.

Plain SUM does not do context transition. This distinction matters inside iterators and calculated columns.

P1 TRICK: What is the output of: x = [1,2,3]; y = x; y.append(4); print(x)? TRICK▼
Answer
Common mistake: thinking y = x creates a copy. It doesn't.
Output: [1, 2, 3, 4]

y = x does NOT create a copy — it creates a second reference to the SAME list object. When you modify y, you're modifying the same underlying list that x points to.

To create an actual copy:
• Shallow copy: y = x.copy() or y = x[:] or y = list(x)
• Deep copy (nested structures): import copy; y = copy.deepcopy(x)
P2 TRICK: What is the output of: print(0.1 + 0.2 == 0.3)? TRICK▼
Answer
Every data analyst using Python needs to know this. It trips up many people.
Output: False

0.1 + 0.2 = 0.30000000000000004 in Python (and most languages). This is due to floating-point binary representation — these decimals can't be represented exactly in base 2.

Solutions for comparisons:
• abs(0.1 + 0.2 - 0.3) < 1e-10 (tolerance check)
• import math; math.isclose(0.1+0.2, 0.3)
• from decimal import Decimal; Decimal('0.1') + Decimal('0.2') (exact)

This matters for data analysis: never use == for float comparisons.

Trick Question
What is the difference between .mean() and .mean(skipna=True)? And what does .mean(skipna=False) do?
.mean() by default has skipna=True — it ignores NaN values and calculates mean of non-null values.

.mean(skipna=False) — if ANY value in the column is NaN, the entire mean returns NaN. This is useful when you want to be strict: a missing value means the calculation shouldn't proceed.

Trick Question
What is the difference between apply(), map(), and applymap()/map() in pandas?
Series.map(func): Applies function element-wise to a Series. Also used for value substitution (pass a dict). Only works on Series.

DataFrame.apply(func): Applies function along an AXIS (by default axis=0 = per column). Returns Series (one value per column) or DataFrame (when function returns Series).

DataFrame.applymap(func) / new name DataFrame.map(func) (pandas 2.1+): Applies function element-wise to every cell in the DataFrame. Slowest — use vectorized operations when possible.
Interview Question
What are lambda functions and when would you use them vs a regular def function?
A lambda is an anonymous, single-expression function: lambda x: x*2. It's syntactic sugar for a simple def.

Use lambda when: The function is simple (one line), used once/inline (passed to map(), filter(), apply(), sorted()), and naming it would add clutter.

Use def when: Logic is complex, function is reused in multiple places, needs documentation/docstring, has error handling, or takes multiple statements. Lambda functions with multiple operations reduce readability — that's a code smell.

Descriptive Statistics — Complete Reference

CONCEPTMilestone 8

Central Tendency

Mean = Σx / n
Median = middle value (sorted)
Mode = most frequent value

Skewed right: Mean > Median > Mode
Skewed left: Mean < Median < Mode
Symmetric: Mean ≈ Median ≈ Mode

Spread & Shape

Variance σ² = Σ(xᵢ - μ)² / N
Sample Variance s² = Σ(xᵢ - x̄)² / (n-1)
Std Dev = √Variance

S1 TRICK: Why does sample variance use (n-1) instead of n in the denominator? What happens if you use n? TRICK▼
Answer — Bessel's Correction
Many people say "I've seen it in formulas" without understanding why. This is a classic stat interview question.
Using n gives the biased estimate — it consistently UNDERESTIMATES the true population variance. This is because the sample mean is calculated FROM the same sample, so the deviations from it are smaller than deviations from the true population mean.

Using (n-1) gives the unbiased estimate — it corrects for the "degree of freedom" used up by estimating the mean from the same data.

In Python: np.std(x, ddof=0) = population std (divides by n), np.std(x, ddof=1) = sample std (divides by n-1). Pandas .std() uses ddof=1 by default.
S2 TRICK: A dataset has mean=100, median=60. Is it skewed? Which direction? Can you tell which way without seeing the data? TRICK▼
Answer
Most people answer correctly but can't fully explain the logic
Right-skewed (positive skew). Mean (100) > Median (60) strongly indicates a long right tail — a few extremely high values are pulling the mean up significantly.

Real-world example: Income data (most people earn modest amounts, but a few billionaires dramatically increase the mean), house prices, insurance claims, social media follower counts.

Rule of thumb: Mean > Median → right skew. Mean < Median → left skew. The larger the gap, the more extreme the skew.

S3 CLASSIC TRICK: The Monty Hall Problem. You pick door 1. Host opens door 3 (goat). Should you switch to door 2? What are the probabilities? TRICK▼
Answer
Counterintuitive answer that even famous mathematicians initially got wrong
YES, always switch! Switching wins 2/3 of the time.

Intuition: When you first pick, P(car behind door 1) = 1/3. P(car behind one of doors 2 or 3) = 2/3. When the host KNOWINGLY opens a goat door, he transfers all that 2/3 probability to door 2. So P(door 2) = 2/3 after switch.

Staying gives 1/3 chance. Switching gives 2/3 chance. If this feels wrong, simulate it: run 10,000 trials in Python — switchers win ~67% of the time.
S4 A medical test is 95% accurate. The disease affects 1% of the population. You test positive. What is the ACTUAL probability you have the disease? Hard▼
Answer — Bayes' Theorem in Action
Almost everyone intuitively answers "95%." The correct answer will shock you.
# Bayes Theorem: P(Disease | Positive) = P(Positive | Disease) × P(Disease) / P(Positive)

P_disease    = 0.01   # 1% prevalence
P_pos_given_disease = 0.95  # true positive rate (sensitivity)
P_pos_given_no_disease = 0.05  # false positive rate (1 - specificity)
P_no_disease = 1 - P_disease  # 0.99

# Total probability of testing positive:
P_positive = (P_pos_given_disease * P_disease + 
              P_pos_given_no_disease * P_no_disease)
# = 0.95 × 0.01 + 0.05 × 0.99 = 0.0095 + 0.0495 = 0.059

P_disease_given_pos = (P_pos_given_disease * P_disease) / P_positive
# = 0.0095 / 0.059 ≈ 0.161 = 16.1%
⚠️ Only 16.1%! Even with a 95% accurate test, low disease prevalence means most positive tests are FALSE positives. This is base rate neglect — critical for medical AI and fraud detection.

S5 TRICK: You run an A/B test and get p=0.048. You declare victory and ship version B. Your manager asks "what's the probability you made the wrong decision?" What do you say? TRICK▼
Answer
Most people say "4.8%." The real answer is much more nuanced and important.
You CANNOT say it's 4.8%. That's the most common misinterpretation of p-values.

The p-value (0.048) means: "If H₀ were true (no real difference), there's a 4.8% chance of seeing this result or more extreme by chance."

It does NOT mean: "there's a 4.8% chance H₀ is true" or "there's a 95.2% chance B is better."

The actual probability of making the wrong decision (False Positive Rate) depends on:
• The prior probability that B is actually better (base rate of winning A/B tests ≈ 10-20%)
• Using Bayes: P(wrong | p=0.048) could be 30-50%+ if base rates are low

Better decision criteria: Effect size (was the lift meaningful?), confidence intervals, business impact, minimum detectable effect — not just p < 0.05.
S6 TRICK: Three A/B tests are run simultaneously on the same website. Each uses α=0.05. What is the probability of at least one false positive? TRICK▼
Answer — Multiple Comparisons Problem
People often assume 3 × 5% = 15%. The real calculation is different.
P(at least one false positive) = 1 − P(no false positives)
= 1 − (0.95)³ = 1 − 0.857 = 14.3%

With 20 simultaneous tests: 1 − 0.95²⁰ = 64% chance of at least one false positive!

Solutions:
• Bonferroni correction: use α/k per test (for 3 tests: α = 0.05/3 = 0.017)
• Benjamini-Hochberg FDR control: less conservative, controls False Discovery Rate
• Run tests sequentially not simultaneously

High R² can be misleading in several ways:

You have a dataset where 10% of values are outliers. You remove them and R² jumps from 0.4 to 0.9. Should you report the cleaned model's results?
Not without disclosure and justification. Removing 10% of data is substantial — this is a major methodological decision that must be:
1. Clearly disclosed in reporting
2. Justified with legitimate reasons (data entry errors, sensor malfunctions, truly anomalous events)
3. Validated that outliers weren't removed just to improve metrics (data dredging)

The outliers might be the most important data points! In fraud detection, churn prediction, or anomaly detection — outliers ARE the signal you're trying to find.
Trick Question
Two variables have a correlation of 0. Does this mean they are independent?
Almost everyone says YES. The correct answer is NO.
NO. Zero correlation (Pearson r = 0) only means no LINEAR relationship.

Perfect example: y = x². If x goes from -3 to +3 symmetrically, r = 0 (no linear relationship) BUT there's a perfect non-linear relationship.

Pearson correlation only measures linear association. Variables can be strongly dependent (quadratic, sinusoidal, etc.) while showing r = 0.

For general independence, use mutual information (information theory) or plot the data — always visualize before reporting correlation.

B2 Build a cohort retention table in Python/SQL from an events table with columns: user_id, event_date, event_type. Hard▼
Answer — Python
import pandas as pd

# Assume df has: user_id, event_date, event_type
df['event_month'] = pd.to_datetime(df['event_date']).dt.to_period('M')

# 1. Get each user's cohort (first activity month)
cohort_df = (df.groupby('user_id')['event_month']
               .min()
               .reset_index()
               .rename(columns={'event_month': 'cohort_month'}))

df = df.merge(cohort_df, on='user_id')

# 2. Calculate months since cohort
df['months_since_join'] = (
    df['event_month'].dt.to_timestamp() - 
    df['cohort_month'].dt.to_timestamp()
).dt.days // 30

# 3. Count unique users per cohort × month offset
cohort_counts = (df.groupby(['cohort_month', 'months_since_join'])
                   ['user_id'].nunique().reset_index())

# 4. Pivot and calculate retention %
cohort_pivot = cohort_counts.pivot(
    index='cohort_month', columns='months_since_join', values='user_id')
cohort_size = cohort_pivot.iloc[:, 0]
retention = cohort_pivot.divide(cohort_size, axis=0).round(3) * 100
print(retention)

Profitability Analysis: Your company's revenue grew 20% last year but profits fell 15%. Diagnose the problem.
Framework — Revenue × Cost decomposition:

1. Revenue quality check: Is the 20% revenue growth from high-margin or low-margin products? Shift in product mix toward cheaper products?
2. COGS analysis: Did unit costs increase? (Raw materials, labor, logistics) — revenue went up but margins compressed?

Case Study Interview
Churn Analysis: Monthly churn went from 2% to 5% over 6 months. How do you analyze and address this?
Step 1 — Diagnose WHO is churning
• Segment by: age of account, plan type, usage level, industry, acquisition channel
• Build a churn cohort: did Jan joiners churn at 5% but June joiners at 10%? (Product experience degraded?)
• Feature usage: are churned users using less of key features?

Step 2 — Diagnose WHY
• Exit surveys, support tickets, NPS comments
• Compare product usage in last 90 days: churned vs retained users
• Correlate churn with timeline: did pricing change? Competitor launched?

Step 3 — Build a Churn Prediction Model
• Features: login frequency, feature usage, support contacts, billing issues
• Model: logistic regression or gradient boosting on historical churners
• Target: intervene with high-probability churners (discount, success call)

KPIs to track: Early churn (first 30 days), involuntary churn (payment failures), voluntary churn rate by segment

General Analytics Trick
You increase sample size from 100 to 10,000. Your p-value drops from 0.3 to 0.001. Did the business effect get bigger?
Most people confuse statistical significance with practical significance
NOT necessarily — and this is a critical distinction.

Statistical significance is highly sensitive to sample size. With 10,000 users, even a 0.1% difference in conversion rate will be "statistically significant" (p < 0.001). But a 0.1% lift may be completely meaningless for business decisions.

The business effect (EFFECT SIZE) doesn't change with sample size. What changes is your ABILITY TO DETECT it.

Always report alongside p-values: Effect size (Cohen's d, relative lift %), confidence intervals, and whether the effect is practically meaningful given business context.
SQL Trick
SELECT * FROM orders LIMIT 10 — will this always return the same 10 rows?
Most people say yes. The actual answer is: it depends and is generally NO.
NO — without ORDER BY, the row order is non-deterministic.

SQL tables are fundamentally unordered sets. The database engine returns rows in whatever order is most efficient (often disk/insertion order, but this can change with database maintenance, parallel execution, index rebuilding, or query plan changes).

Rule: Always use ORDER BY if you care about which rows are returned by LIMIT. LIMIT without ORDER BY = random sample, not consistent results.
Statistics Trick
Average salary at a company is $80K. You hire one CEO at $2M. The average is now $95K. Is the company "doing better" on compensation?
The average is misleading — the MEDIAN barely moved.

This is why median is superior for salary/compensation reporting. One extreme value (the CEO) pulled the mean dramatically without affecting typical employee pay at all.

In real analytics: when presenting compensation data to employees or stakeholders, always use MEDIAN. When calculating total payroll budget, use SUM. Mean is appropriate only when the distribution is symmetric with no extreme outliers.
Python Trick
df.merge(df2, on='id') returned MORE rows than either input DataFrame. Is this a bug?
Not a bug — it's a many-to-many join (Cartesian product on duplicates).

If df has 3 rows with id=5 and df2 has 4 rows with id=5, the merge produces 3×4=12 rows for id=5.

Always validate merge results:
• df.merge(df2, on='id', validate='1:1') — raises error if not one-to-one
• validate='1:m' or 'm:1' for one-to-many
• Always check len(result) vs len(df) after merging
Excel Trick
Your SUMIFS formula returns 0 even though matching data clearly exists. What are all the possible reasons?
1. Number-text mismatch: Sum range has numbers stored as text (can't be summed). Look for green triangles. Fix: multiply by 1 or use VALUE().
2. Criteria type mismatch: Criteria range has numbers but you're comparing to text "5" instead of 5 (or vice versa).
3. Trailing spaces: "Smith " ≠ "Smith". Use TRIM() on criteria.
4. Range sizes don't match: sum_range and criteria_range must be the same size (same number of rows).
5. Wildcard not quoted: Use criteria "North*" not North* (wildcard must be in a text string).
6. Date comparison issue: Dates stored as text vs date serial numbers.
Power BI Trick
You create a measure and it shows the correct value in a bar chart but shows BLANK in a table visual. Why?
The measure returns BLANK() for rows with no data, and tables show all rows while bar charts aggregate.

When a table contains a dimension with values that have no matching fact data, the measure evaluates to BLANK. Bar charts typically skip blanks or show zero. A table shows every row in the visible context including those with blanks.

Fix options:
• Add + 0 to convert BLANK to 0: [Total Revenue] + 0
• Use IF(ISBLANK([Total Revenue]), 0, [Total Revenue])
• Filter out the rows with no data using visual-level filters
